SOC Homelab

A multi-VM Security Operations environment built on Proxmox VE for hands-on detection, log analysis, and incident response practice.

Proxmox VE Splunk Enterprise Sysmon Windows 10 Ubuntu Server rsyslog

Objective

Build a realistic SOC environment from scratch to practice the skills required for a Security Operations Center Analyst role — log ingestion, endpoint telemetry, alert triage, and detection engineering — without relying on simulated cloud labs.

Architecture

The lab runs on a dedicated physical machine using Proxmox VE as the hypervisor, with VMs segmented into a isolated lab network separate from management traffic.

Windows 10 Endpoint

Primary target machine. Generates Windows Event Logs and Sysmon telemetry forwarded to Splunk via Universal Forwarder.

Ubuntu Server

Linux endpoint configured with rsyslog to forward syslog data into Splunk for centralized Linux visibility.

Splunk Enterprise

Central SIEM receiving logs from all endpoints. Used for log search, investigation, and detection rule development via SPL.

Kali Linux (In Progress)

Attacker VM being deployed to simulate adversary techniques and validate detection coverage against generated endpoint telemetry.

What Was Built

  • Network Segmentation — Lab traffic isolated from management access via separate Proxmox network segments, preventing cross-contamination between attacker and management interfaces.
  • Splunk SIEM Deployment — Splunk Enterprise installed and configured with Universal Forwarder on the Windows endpoint to centralize log ingestion. SPL queries used to validate and search endpoint telemetry.
  • Sysmon Tuning — Deployed Sysmon with the SwiftOnSecurity configuration to capture process creation (Event ID 1), network connections (Event ID 3), and file I/O events (Event ID 11) for high-fidelity endpoint visibility.
  • Linux Log Forwarding — Configured rsyslog on Ubuntu Server to forward syslog data into Splunk, providing cross-platform log coverage.
  • Windows Event Log Coverage — Enabled and verified key Windows Event IDs flowing into Splunk alongside Sysmon data for layered endpoint visibility.

Key Troubleshooting — Hostname Mismatch

During initial setup, Windows endpoint data was not appearing in Splunk as expected. Rather than redeploying the forwarder immediately, a structured troubleshooting approach was used:

1
Ruled out network connectivity

Verified the Universal Forwarder could reach the Splunk indexer on port 9997. Connectivity was confirmed — not the issue.

2
Ruled out configuration errors

Checked inputs.conf and outputs.conf on the forwarder. Configuration was correct — not the issue.

3
Identified root cause — hostname mismatch

Data was arriving in Splunk but indexed under an unexpected hostname. The Windows VM hostname did not match the expected value in the SPL search filter, causing the data to appear missing when it was actually present under a different host value.

This troubleshooting process mirrors real SOC investigation methodology — rule out the obvious before assuming the complex.
